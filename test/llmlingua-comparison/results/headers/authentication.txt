>
> The iGaming Platform has migrated from Supabase Auth to a unified custom authentication system. This affects both Operator Web (backoffice) and Player Web (casino) applications.

### What Changed
| Aspect | Previous (Supabase Auth) | Current (Custom Auth) |
|--------|--------------------------|----------------------|
| **API Endpoint** | `/auth/v1/token` | `/functions/v1/auth/*` |
| **Database Table** | `auth.users` | `player.player_accounts` |
| **Token Type** | Supabase JWT | Custom JWT (HMAC-SHA256) |
| **Admin Detection** | `app_metadata.role = "admin"` | `is_admin = TRUE` column |
| *…

### Breaking Changes for Existing Deployments
1. **API Endpoints**: All auth calls must use `/functions/v1/auth/*` instead of `/auth/v1/*`
2. **User Data**: Migrate users from `auth.users` to `player.player_accounts`
3. **Admin Access**: Update admin detection from `app_metadata.role` to `is_admin` column
4. **Token Validation**: Update middleware to validate custom JWTs

### Migration Steps
1. Run database migrations to create `player.player_accounts` and related tables
2. Migrate existing user data (if applicable)
3. Update frontend auth stores to use new API endpoints
4. Update middleware to validate custom JWT tokens
5. Test authentication flows end-to-end

For detailed migration guidance, contact the development team.

## Authentication Methods

### 1. Email/Password Authentication
Traditional credential-based authentication with email and password.

**Flow:**

1. User registers with email and password
2. Password hashed with bcrypt (cost factor 10)
3. Email verification code sent (15-minute expiry)
4. User verifies email to activate account
5. User can login with email + password

**Security Features:**

- Bcrypt password hashing
- Email verification required
- Failed login…

### 2. Phone/Password Authentication
Alternative credential-based authentication using phone number.

**Flow:**

1. User registers with E.164 phone number and password
2. SMS verification code sent (5-minute expiry)
3. User verifies phone to activate account
4. User can login with phone + password

**Phone Format:** E.164 international format (e.g., `+14155552671`)

### 3. Magic Link (Passwordless)
Email-based passwordless authentication.

**Flow:**

1. User requests magic link via email
2. Unique token generated (UUID) with 15-minute expiry
3. Token stored in database with single-use enforcement
4. Email sent with authentication link
5. User clicks link, token verified and consumed
6. Session created automatically

**Security Features:**

- Single-use tokens (atomic verify-and-consume)
- 15…

### 4. Social Authentication
OAuth 2.0 authentication with Apple and Google.

**Supported Providers:**

- Apple Sign In
- Google Sign In

**Flow:**

1. Client obtains OAuth token from provider
2. Token validated with provider
3. User profile retrieved
4. Account created or linked to existing account
5. Session created

**Account Linking:** Social auth can be linked to existing accounts.

### 5. Two-Factor Authentication (2FA)
See dedicated [Two-Factor Authentication](#two-factor-authentication) section.

## Two-Factor Authentication

### TOTP (Authenticator App)
**RFC 6238 Compliant Implementation:**

- Algorithm: HMAC-SHA1
- Digits: 6
- Time Step: 30 seconds
- Window: ±1 step (90 seconds total)
- Secret Size: 160 bits (20 bytes)

**2FA Secret Encryption (Updated 2026-02-04)**

TOTP secrets are encrypted using PostgreSQL pgcrypto with the `ENCRYPTION_KEY` environment variable:

```sql
-- Encrypt a secret (stored as base64 in database)
SELECT public.encryp…

### Backup Codes
Generated during TOTP setup:

- **Count:** 10 codes
- **Format:** `XXXX-XXXX` (8 alphanumeric characters)
- **Entropy:** 40 bits per code
- **Security:** HMAC-SHA256 hashed before storage
- **Single-Use:** Marked as used with timestamp
- **Regeneration:** Can generate new set (invalidates old codes)

### SMS OTP
6-digit OTP sent via SMS:

- **Expiry:** 5 minutes
- **Rate Limit:** 5 attempts per 15 minutes (when enforced)
- **Security:** Rejection sampling prevents modulo bias
- **Atomic Attempt Tracking:** Redis Lua script prevents race conditions

### Email OTP
6-digit OTP sent via email:

- **Expiry:** 10 minutes
- **Rate Limit:** 5 attempts per 15 minutes (when enforced)
- **Same security model as SMS OTP**

### 2FA Login Flow
```text
1. User enters email + password
2. System validates credentials
3. System detects 2FA enabled
4. Response includes pendingSessionToken and requires2FA=true
5. User enters TOTP/SMS/Email code or backup code
6. System verifies code
7. System upgrades pending session to full session
8. Returns sessionToken + refreshToken
```text

## Security Considerations

### Token Security
- **Generation:** `crypto.randomUUID()` provides 128-bit entropy
- **Hashing:** HMAC-SHA256 with environment-configured secret (minimum 32 characters)
- **Storage:** Never stored in plaintext, always hashed
- **Transmission:** Only over TLS/HTTPS
- **Rotation:** Refresh tokens rotated on every use

### Password Security
- **Hashing:** Bcrypt via PostgreSQL `pgcrypto` extension
- **Cost Factor:** 10 (configurable)
- **Verification:** Database-side via `verify_password()` function
- **Requirements:**
  - Minimum 8 characters
  - At least one uppercase letter
  - At least one lowercase letter
  - At least one number
  - At least one special character

### Rate Limiting Configuration
**Note:** Rate limiters are instantiated but NOT currently enforced (Issue GAP-004 HIGH).

**Configured Limits:**

- Login: 5 attempts per 15 minutes (per IP)
- 2FA Verification: 5 attempts per 15 minutes (per session)
- Password Reset: 3 requests per 60 minutes (per email)
- Registration: 3 attempts per 60 minutes (per IP)
- Magic Link: 3 requests per 60 minutes (per email)

**Enforcement:** CRIT…

### Account Lockout Policy
**Note:** Currently NOT implemented (Issue GAP-003 MEDIUM).

**Recommended Policy:**

- Lock account after 10 consecutive failed login attempts
- Require email verification to unlock
- Send security alert email on lockout
- Track lockout events in audit log
